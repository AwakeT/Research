# 家庭语义记忆系统——分层架构设计

## 0. 前提与方向确认

**已确定的演进方向**：以 VLM 在线拓扑语义建图为核心采集路径，放弃以几何分区算法为主线的演进。

**需要解决的两个目标**：
1. 机器内部——对家庭场景"记住、理解、推理"
2. 用户外部——对家庭空间"看懂、信任、可纠错"

**已有文档（home-memory-representation-final.md）的问题**：三层结构方向正确，但层内职责混杂：底层把静态结构与动态事件混放；中间层把存储模块与路由机制并列；展示层没有说清由哪层数据驱动。本文档在此基础上重新厘清。

**本次补充优化重点**：
- 补齐对象身份模型，区分锚点物体与可移动物体
- 明确地图版本一致性约束，避免重定位/漂移污染记忆
- 让用户纠错具备审计与回滚能力，而不是只改展示结果
- 澄清 L3 是否允许派生索引，避免实现时变成隐性存储层
- 为 L4 增加"何时可信、何时降级展示"的边界

---

## 1. 重新定义四层结构

原有三层不足以清晰表达各模块的时效性、读写频率和职责边界。建议扩展为**四层**：

```
┌─────────────────────────────────────────┐
│           L4：用户展示层                 │  ← `用户可见`
├─────────────────────────────────────────┤
│           L3：记忆服务层                 │  ← 任务/LLM 消费
├─────────────────────────────────────────┤
│           L2：动态事件层                 │  ← 高频写入、短生命周期
├─────────────────────────────────────────┤
│           L1：语义结构层                 │  ← 低频写入、长生命周期
└─────────────────────────────────────────┘
         ↑ VLM 在线拓扑建图持续写入 L1 + L2
```

关键区分：**L1 是"这个家长什么样"，L2 是"这个家最近发生了什么"**，两者读写频率和时效性完全不同，不应混在同一层。

---

## 2. L1：语义结构层（Semantic Structure Layer）

**定位**：家庭空间的长期稳定骨架，低频写入，长期有效。

**写入来源**：VLM 在线拓扑建图的聚合结果；用户主动纠错；多次观测的统计稳定结论。

### 2.1 核心数据结构

#### 房间节点（Room Node）
```json
{
  "id": "room_001",
  "type": "living_room",
  "neighbors": ["room_002", "room_003"],
  "bbox": { "cx": 3.2, "cy": 6.1, "w": 5.0, "h": 4.0 },
  "last_confirmed": "2026-03-28T10:00:00Z"
}
```

> `bbox` 来自 SLAM 或 VLM 推断，一旦由多次观测稳定确认后冻结，不随单次观测更新。

#### 区域节点（Zone Node）
```json
{
  "id": "zone_sofa_area",
  "parent_room": "room_001",
  "bbox": { "cx": 2.1, "cy": 5.8, "w": 2.5, "h": 1.8 },
  "anchor_objects": ["obj_sofa_001", "obj_coffee_table_001"],
  "remembered_objects": [
    { "category": "remote_control", "evidence_count": 8 },
    { "category": "pillow", "evidence_count": 5 }
  ]
}
```

> `bbox` 由单次观测中多个锚点物品的位置关系共同推断得出，基本可信；一旦确认后冻结，仅当区域内锚点物品集合发生变化时重新计算。
>
> `remembered_objects` 是该区域的持久记忆物品表，由 L2 观测事件沉淀写入：出现次数超过阈值则加入，长期未观测到则 `evidence_count` 归零后删除。L2 Observation Event 中的 `new_objects` / `unseen_objects` 均以此表为基准计算，而非相对上一次单次观测。

#### 锚点对象节点（Anchor Object Node）
```json
{
  "id": "obj_sofa_001",
  "category": "sofa",
  "canonical_room": "room_001",
  "canonical_zone": "zone_sofa_area",
  "bbox": { "cx": 2.1, "cy": 5.8, "w": 2.5, "h": 1.8 },
  "position_relative": "靠近西墙窗边"
}
```

> 锚点对象归属于某个区域（多个锚点对象可共属同一区域）。`position_relative` 为辅助描述，供 LLM 消费；区域几何边界由 Zone Node 的 `bbox` 承载，不在对象节点重复存储。

#### 可移动对象记忆节点（Movable Object Memory Node）
```json
{
  "id": "obj_medicine_box",
  "category": "medicine_box",
  "habitual_locations": [
    {
      "room": "room_004",
      "evidence_count": 12
    },
    {
      "room": "room_005",
      "evidence_count": 6
    }
  ],
  "last_confirmed_location": {
    "room": "room_005",
    "source": "vlm"
  },
  "last_confirmed_at": "2026-03-28T20:15:00Z"
}
```

> 可移动对象位置完全由 VLM 观测驱动，不设用户纠错环节。位置粒度仅到房间级，不下沉到区域（可移动对象不锚定区域）。

### 2.2 设计要点

- **L1 只存节点与直接归属关系**：对象→区域、区域→房间、房间→邻接房间；复杂语义关系（"常见于"、"习惯性出现在"、"不应出现在"等）由 L3 在查询时从 L1+L2 推导，不预先写入 L1
- **bbox 一旦确认即冻结**：区域 bbox 由单次多锚点观测推断，房间 bbox 由多次观测稳定后确认；两者均不随单次新观测更新，仅在结构发生显著变化时重算
- **节点不随单次观测失效**：只有累计多次观测触发沉淀或结构变化才更新 L1
- **L1 记录的是长期语义事实，不直接吞并原始观测噪声**：任何结构更新都应来自 L2 的沉淀或明确初始化流程

## 3. L2：动态事件层（Dynamic Event Layer）

**定位**：记录高频、短生命周期的观测事件和状态变化，是"最近发生什么"的真实来源。

**写入来源**：VLM 每次采样的原始输出。

**保留策略**：按时间窗口保留（如最近 7 天详细，更早的压缩摘要），不无限积累。

> 前提约束：机器人开始运动时地图已保证一致，无需在事件中额外携带地图版本或位姿信息。

### 3.1 核心数据结构

#### 观测事件（Observation Event）
```json
{
  "event_id": "obs_20260328_145201",
  "timestamp": "2026-03-28T14:52:01Z",
  "zone": "zone_sofa_area",
  "vlm_output": {
    "visible_objects": ["remote_control", "coffee_cup", "newspaper"],
    "new_objects": ["bottle", "hat"],
    "unseen_objects": ["medicine_box"]
  }
}
```

> 记录机器人在某区域的一次 VLM 采样结果。`visible_objects` 是本次可见物品全集；`new_objects` 是相比上次观测新出现的；`unseen_objects` 是上次可见但本次消失的。供沉淀机制统计对象出现频率和变化。

#### 对象状态事件（Object State Event）
```json
{
  "event_id": "obj_state_20260328_093011",
  "timestamp": "2026-03-28T09:30:11Z",
  "object_category": "medicine_box",
  "last_seen_room": "room_004"
}
```

> 记录某类可移动对象最近一次被观测到的房间。由观测事件中的 `visible_objects` 派生写入，作为沉淀机制更新 `habitual_locations` 的依据。

#### 空间切换事件（Room Transition Event）
```json
{
  "event_id": "trans_20260328_093500",
  "timestamp": "2026-03-28T09:35:00Z",
  "from_room": "room_001",
  "to_room": "room_002"
}
```

> 记录机器人从一个房间进入另一个房间的时刻。多次重复的切换对用于强化 L1 中两个房间的 `neighbors` 邻接关系。

### 3.2 L2 → L1 的沉淀机制

L2 事件不直接暴露给上层消费，而是通过定期沉淀更新 L1：

- **区域物品记忆沉淀**：某对象在某区域的观测事件中多次出现（`evidence_count` 超过阈值）→ 加入 L1 Zone Node 的 `remembered_objects`；长期出现在 `unseen_objects` 中导致 `evidence_count` 归零 → 从 `remembered_objects` 删除
- **可移动对象位置沉淀**：某对象状态事件在某房间累计出现 N 次 → 更新 L1 可移动对象 `habitual_locations` 分布
- **新房间发现**：多次未匹配已有房间的观测 → 触发 L1 新节点创建
- **邻接关系强化**：空间切换事件多次重复 → 强化 L1 `neighbors` 关系
- **遗忘衰减**：L1 可移动对象长期无 L2 证据支撑 → `evidence_count` 自然下降，不直接删除节点

## 4. L3：记忆服务层（Memory Service Layer）

**定位**：不作为事实源，负责把 L1+L2 转化为三类核心任务（找人、找物、巡逻异常检测）可直接消费的形态。它是"翻译层"，必要时允许存在可重建的派生索引。

**核心职责**：按任务类型按需组装，不预先把所有视图写死；允许为性能维护轻量索引。

### 4.0 轻量派生索引（建议有，但不是事实源）

L3 可以维护少量**可重建、可失效**的派生结果：

- `room_summary_cache`：各房间当前状态摘要，加速巡逻异常检测的基线比对
- `object_location_index`：可移动对象的最近位置索引，加速找物路由
- `zone_snapshot_baseline`：各区域 `remembered_objects` 的历史基线快照，供异常检测差分

这些索引的作用是加速查询，不是事实源，关键查询仍回到 L1/L2 校验。

### 4.0b 语义关系推导（Semantic Relation Inference）

L1 只存节点与直接归属关系，复杂语义关系由 L3 在查询时按需推导：

| 关系 | 推导来源 | 示例 |
|---|---|---|
| `habitually_found_in` | L1 可移动对象 `habitual_locations` | 药箱 → 主卫（60%）/ 次卫（30%） |
| `not_expected_in` | 常识规则 + L2 历史缺席统计 | 清洁剂 → 儿童房 |
| `used_for` | VLM 描述聚合 | 阳台柜 → 收纳清洁工具 |

这些关系不写入 L1，不持久化存储，查询用完即丢。

### 4.1 任务场景服务接口

#### A. 找物（Object Search）

输入：目标对象类别（如"药箱"）
输出：优先探索的区域有序列表 + 每个区域的历史概率

```
任务："找药箱"
L3 组装输出：
  候选房间（按 habitual_locations 概率排序）：
    1. 主卫（60%，上次观测：2026-03-28 09:30）
    2. 次卫（30%，上次观测：2026-03-27 14:10）
  区域内优先探索区域：
    主卫 → zone_mirror_cabinet（remembered_objects 含 medicine_box）
  排除房间：客厅、卧室（历史概率 < 5%）
```

推导来源：L1 `habitual_locations` + L1 Zone `remembered_objects` + L2 最近对象状态事件

#### B. 找人（Person Search）

输入：目标人物（如"老人"）或无具体目标（全局搜索）
输出：按历史活动频率排序的房间列表 + 最近观测时间

```
任务："找老人"
L3 组装输出：
  高频活动区域（按 L2 历史观测统计）：
    1. 客厅沙发区（最近见于：2026-03-31 08:45）
    2. 餐厅（最近见于：2026-03-31 07:30）
  当前未覆盖房间：书房、阳台（今日尚未观测）
  建议起始房间：客厅（最近活动 + 历史最高频）
```

推导来源：L2 观测事件中的 `visible_objects`（含人物类别）按区域/时间聚合统计

#### C. 巡逻异常检测（Patrol Anomaly Detection）

输入：当次巡逻各区域的观测事件
输出：与基线偏差显著的区域 + 异常描述

```
巡逻结果（当次 vs 基线对比）：
  客厅茶几区：
    新增物品：bottle, hat（不在 remembered_objects 中）
    消失物品：newspaper（长期常见，本次 unseen）
    → 异常等级：低（临时物品变化）
  主卫：
    medicine_box 未见（历史常见，连续 3 次 unseen）
    → 异常等级：中（可能已被移走，触发对象状态事件更新）
  儿童房：
    检测到 cleaning_agent（not_expected_in 规则命中）
    → 异常等级：高（安全规则违反，需告警）
```

推导来源：L2 当次观测事件（`new_objects`/`unseen_objects`）vs L1 Zone `remembered_objects` 基线 + `not_expected_in` 语义规则

### 4.2 与 L1/L2 的关系

```
L1 (语义结构) ──┐
                ├──→ L3 记忆服务层 ──→ 找物 / 找人 / 巡逻异常检测
L2 (动态事件) ──┘
```

L3 保持接口无状态；派生索引可重建、可丢弃，不成为隐藏的第三份事实存储。

## 5. L4：用户展示层（User Presentation Layer）

**定位**：由 L3 驱动生成，面向用户可理解性设计，不直接暴露 L1/L2 的数据结构。

**核心原则**：用户要的是"看懂这个家"，不是"看懂系统内部建模"。

### 5.1 三个展示视图

#### A. 房间总览视图（主视图）

由 L1 房间节点 + 邻接关系直接驱动，展示：
- 家里有哪些房间（按 L1 房间节点）
- 房间之间如何相连（按 `adjacent_to` 关系）
- 每个房间的类型标签和主要语义
- 每个房间的 1-3 个关键高频物品

形式：卡片列表 + 关系连线（不是坐标图，是语义关系图）

#### B. 语义布局草图（地图视图）

**重要**：这不是几何地图，是由 L1 关系约束驱动的生成式草图。

生成输入：
- 房间邻接关系（哪些房间相连）
- 房间大小关系（L1 记录的相对面积描述）
- 关键连接点（入口、走廊）

生成原则：
- **只画相对关系，不画绝对坐标**
- **不确定区域弱化处理**，不硬画精确边界
- **允许"大致正确、布局合理"**，不追求工程精度
- 用户看到的是语义示意图，不是 SLAM 栅格图

#### C. 房间详情视图（点击展开）

由 L1 对象节点 + L2 最近事件驱动，展示：
- 该房间内的常见家具和高频物品（来自 L1）
- 最近观测到的状态变化（来自 L2）
- 对象置信度（系统推断 vs 用户确认）
- 用户纠错入口

### 5.2 展示可信度与降级规则

为了保证"用户看懂、信任、可纠错"，L4 不能在证据不足时强行画满：

- 当房间邻接关系置信度不足时，只展示列表，不展示连线图
- 当布局约束不足时，语义草图应降级为"信息不足，暂不生成示意图"
- 展示中要显式区分"用户确认"、"系统推断"、"最近观测"
- 对于最近变化，优先展示时间和证据来源，避免把单次观测说成长期事实

### 5.3 用户纠错闭环

用户纠错不能只停在 L4（只改展示而不改底层），必须回写到 L1：

```
用户操作："药箱不在主卫，现在放次卫了"
  → L4 更新展示
  → 写入 L2：user_correction_event（保留旧值、新值、时间、原因）
  → 写入 L1：obj_medicine_box.last_confirmed_location = {
                room: "room_005",
                zone: "zone_bathroom_secondary",
                source: "user"
              }
              obj_medicine_box.user_confirmed = true
              obj_medicine_box.last_confirmed_at = now()
              obj_medicine_box.habitual_locations 中对应次卫权重上调
  → 触发 L2 降低该对象旧常驻位置历史置信度，而不是直接抹除历史
```

用户确认的 L1 条目在沉淀机制中具有最高优先级，不会被 VLM 观测统计直接覆盖；但旧历史仍保留在 L2 中，便于审计与回滚。

## 6. VLM 在线拓扑建图与四层结构的关系

PDF 中提出的 VLM 在线拓扑建图是**整套系统的数据采集入口**，对应四层结构中的写入来源：

```
机器人执行导航动作
  │
  ├─ 满足采样条件（位移/角度/时间间隔）
  │
  ├─ 采集 RGB + 当前位姿 + map_epoch/topology_version
  │
  ├─ 图像 + 约束式提示词 → VLM
  │
  ├─ VLM 输出：{房间类型, 物品列表, 显著变化}
  │
  ├─ 判断"位姿邻近性 + 语义相似性 + 当前 map_epoch" → 归属已有房间？
  │     ├─ 是 → 写入 L2 观测事件（更新已有节点统计）
  │     └─ 否 → 写入 L2 观测事件（标记为"候选新房间"）
  │
  ├─ 发生空间切换 → 写入 L2 房间切换事件 → 创建/强化 L1 邻接关系
  │
  ├─ 发生重定位/重建图 → 开启新的 map_epoch
  │
  └─ 定期触发 L2 → L1 沉淀（更新语义结构层）
```

这套流程天然在线、增量更新，不需要任务结束后再统一离线处理。

## 7. 各层的数据生命周期对比

| 维度 | L1 语义结构层 | L2 动态事件层 | L3 记忆服务层 | L4 用户展示层 |
|---|---|---|---|---|
| 写入频率 | 低（沉淀触发） | 高（每次采样） | 低到中（派生索引可刷新） | 低（用户交互时） |
| 读取频率 | 中 | 中 | 高 | 高 |
| 生命周期 | 长期（月/年级别） | 短期（7天详细+压缩摘要） | 可重建、可失效 | 随 L1 更新 |
| 时效性要求 | 低 | 高 | 中 | 低 |
| 用户可见 | 否（底层） | 否（底层） | 否（服务层/索引层） | 是 |
| 用户可纠错 | 是（回写入口） | 否 | 否 | 是（触发 L1 回写） |

## 8. 与当前技术路线的对接

### 8.1 几何底层保留为基础设施

几何地图（SLAM 栅格 / 拓扑导航图）不消失，降为**底层运动基础设施**：
- 负责避障、路径规划、到达性计算、回充
- 不再是家庭记忆的主表示

L1 中的 `position_hint` 字段可以与几何坐标软绑定，但不强依赖——语义关系即使在坐标漂移后仍然有效。

### 8.2 已有 U-Net 分区模型的位置

U-Net 分区结果可以作为 L1 初始化的**冷启动输入**：
- 新环境首次建图时，用 U-Net 生成初始房间轮廓作为 L1 的种子
- 后续靠 VLM 在线建图持续修正和补充
- 长期不再依赖 U-Net 分区维持地图精度

### 8.3 房间语义分类器的位置

当前基于最大内切圆采集4张图的分类器，可以演进为 VLM prompt 中的结构化约束，而不再是独立部署的分类 head：
- 减少维护两套模型的成本
- VLM 返回的 `room_type` 直接对应 L1 `type` 字段
- 多帧融合的一致性判断在 L2 → L1 沉淀机制中完成

## 9. 已知风险与应对

| 风险 | 影响层 | 应对策略 |
|---|---|---|
| VLM 对同一房间判断不稳定 | L2 → L1 沉淀 | 多次观测置信度阈值机制；用户确认优先 |
| 房间切换检测误触发 | L2 空间切换事件 | 双判据（位姿 + 语义）；切换事件需 N 帧稳定确认 |
| 对象识别漏检导致 L1 偏差 | L1 对象节点 | L1 不因单次漏检删除；基于统计频率而非单次事件 |
| 可移动物体被错误当成常驻锚点 | L1 / L2 | schema 显式区分 `anchor` 和 `movable`；可移动物体默认沉淀为位置分布 |
| 跨地图版本误聚合导致结构污染 | L2 → L1 | 每条事件带 `map_epoch/topology_version`；跨 epoch 只能语义对齐后再融合 |
| 用户纠错未回写到 L1 | L4 / L1 | 纠错入口必须触发 L1 写入，L4 不单独维护状态 |
| 用户纠错不可追溯，后续无法解释/回滚 | L2 / L1 | 纠错先写 `user_correction_event`，再更新 L1 |
| L2 数据膨胀 | L2 | 固定时间窗口保留 + 压缩摘要；不无限积累 |
| 几何坐标漂移导致 L1 失效 | L1 | L1 核心是语义关系，不强依赖绝对坐标；`position_hint` 仅辅助 |
| L3 派生索引过期，造成答复与底层事实不一致 | L3 | 派生索引只作加速，不作事实源；关键查询回源 L1/L2 校验 |
| 生成式草图"画得合理但不真实" | L4 | 明确标注"示意图"；不确定区域弱化；用户可纠错 |

## 10. MVP 落地路径

### 阶段一：最小可用记忆系统
- **L1**：房间节点 + 邻接关系（仅拓扑骨架）
- **L2**：观测事件流（带 `map_epoch/topology_version`，只存，不沉淀）
- **L3**：Room Summary 生成（文本摘要，供 LLM 消费）
- **L4**：房间卡片列表（文字形式，无草图）

验证目标：VLM 在线建图能否稳定识别和维护 N 个房间的拓扑关系

### 阶段二：对象记忆和动态更新
- **L1**：加入对象节点 + 功能区节点，并显式区分 `anchor` / `movable`
- **L2**：加入对象状态事件 + 用户纠错事件 + L2→L1 沉淀机制
- **L3**：加入 Object Memory Card + 检索路由
- **L4**：房间详情展开视图 + 用户纠错入口

验证目标：能否回答"某物通常在哪"；用户纠错能否正确回写

### 阶段三：完整服务能力
- **L3**：加入执行抽象（Task Abstraction）和轻量派生索引
- **L4**：加入带可信度约束的生成式布局草图（由 L1 关系约束驱动）
- 打通与下游任务规划器的接口

验证目标：系统能否支撑完整的"找物 → 解释 → 执行 → 用户确认"闭环

---

## 11. 一句话总结

**以 VLM 在线拓扑建图为采集入口，L1 语义结构层存"这个家长什么样"，L2 动态事件层存"最近发生了什么且在什么地图版本下发生"，L3 记忆服务层按需把这两者翻译成任务/LLM 可消费的形态，L4 用户展示层让用户看懂、信任并可以纠错——再配上对象身份区分、纠错审计和可信度降级机制，这四层才真正具备可落地性。**

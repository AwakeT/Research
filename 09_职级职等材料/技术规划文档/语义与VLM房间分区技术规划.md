# 语义与VLM房间分区技术规划

## 1. 背景与目标

### 1.1 业务背景

家庭服务机器人在执行导航、找物、巡逻等任务时，需要对室内环境具备房间级别的语义理解能力。当前系统仅有基于ASM（Active Semantic Map）构建的障碍地图，缺少房间分区与语义标注信息，无法支撑"去厨房拿水杯"、"找药箱"等高层任务指令的理解与执行。

### 1.2 核心目标

构建一套从障碍地图到语义房间级地图的完整技术链路，实现：

1. **几何分区**：将2D障碍地图自动划分为若干独立房间区域
2. **语义标注**：识别每个区域的房间类型（客厅、卧室、厨房、卫生间等）
3. **语义记忆**：构建包含房间拓扑、物品位置、状态变化的分层语义记忆系统
4. **任务支撑**：为VLN导航、找物、找人、巡逻异常检测等下游任务提供语义上下文

### 1.3 技术约束

- 端侧部署，需考虑计算资源限制（CPU推理优先）
- 需适配国内实际家居场景（与国外仿真场景存在差异）
- 地图来源为ASM在线构建的障碍地图，非标准户型图

## 2. 技术路线演进

整体技术路线经历了三个阶段的探索与演进，从纯VLM方案逐步收敛为"传统几何分区 + VLM语义标注"的混合架构，最终延伸至深度学习分区和去几何地图化的语义拓扑方案。

### 2.1 阶段一：纯VLM直接分区（已验证不可行）

**技术思路**：将ASM地图直接送入VLM，让模型一步完成分区与语义标注。

**验证方案**：

- **方案A - 单张ASM地图推理**：将带物品标签的俯视图送入VLM，要求以bbox+label的JSON格式返回房间分区结果
- **方案B - ASM地图+机器人位置+第一视角RGB**：多模态融合输入，尝试结合地图全局信息与第一视角局部信息

**测试覆盖的模型**：Qwen3-VL-8B / 32B / 30B-A3B / 235B-A22B、GLM4.5v

**验证结论**：

- VLM无法直接给出精准的分区边界（像素级bbox），输出结果波动大、一致性差
- VLM擅长给出分区指示及房间语义（房间类型判断准确率高），但不擅长几何定位
- 多模态融合输入（地图+RGB+位置）并未显著提升分区精度，8B/30B-A3B等小模型仅能识别出当前所在单个房间

**关键结论**：VLM定位于语义判断角色，不承担几何分区职责。

### 2.2 阶段二：传统几何分区 + VLM语义标注（当前主线方案）

**技术思路**：传统CV算法负责几何分区，VLM负责语义推理，各取所长。

**整体流程**：

1. 基于navmesh+glb文件在habitatsim仿真环境中构建障碍地图
2. 使用CW分区算法（距离变换+分水岭）对裁剪后的障碍地图进行几何分区
3. 提取各分区最大内切圆圆心，转换像素坐标为habitat坐标
4. 将agent传送至各点位采集前后左右四方向RGB图像
5. 送入VLM进行区域类型推理
6. 基于推理结果将类型相同且邻接的区域合并

**参考论文**：SeLRoS（Semantic Layering in Room Segmentation via LLMs, arXiv:2403.12920），在交并比实验中达到约70%准确率（IoU 69.98, MSIoU 57.73）。

### 2.3 阶段三：深度学习分区（当前推进方向）

**技术思路**：基于U-Net训练地图美化与分区模型，摆脱传统几何算法对参数调优的依赖。

**两阶段流程**：

1. **地图美化阶段**：利用带障碍地图和无障碍目标图训练地图美化模型，输出清洁户型图
2. **地图分区阶段**：基于人工标注的房间分区数据训练多头U-Net，预测房间边界（boundary）、房间核心（seed）、室内区域（inside），通过后处理生成带房间实例编号的分区结果图

### 2.4 阶段四：去几何地图化分区（前瞻构思）

**技术思路**：不依赖完整SLAM管线和精确地图，仅通过RGB图像+粗定位数据，在线构建语义拓扑图。

**核心输出**：房间级语义节点 + 房间间连通关系 + 房间内物品语义 + 物品位置信息

## 3. 核心技术模块详细设计

### 3.1 几何分区算法

#### 3.1.1 算法流程

```
输入障碍地图 (ObstacleMap)
    |
    v
prepare_partition_map（旋转 + 裁剪黑边 + 垂直翻转）
    |
    v
_preprocess_map（二值化 / 提取墙体 / 墙体闭运算 / 生成自由空间 / 开运算去噪 / 计算distance_map）
    |
    v
_initial_segmentation（距离变换 + 高斯平滑 + 局部极大值种子 + watershed）
    |                                    |
    v（种子足够）                           v（种子不足）
  watershed分水岭分割                   连通域分析兜底
    |
    v
_expand_regions（区域膨胀扩展到边界）
    |
    v
_remove_tiny_regions + _filter_abnormal_regions（删小区域 / 过滤异常大区域 / 边界异常区域）
    |
    v
_merge_regions（按面积 / 狭长程度 / 公共边界长度启发式合并）
    |
    v
_split_regions_by_narrow_connections（对大区域做窄连接切分后处理）
    |
    v
_generate_result（生成最终 result_map / rooms / statistics）
```

#### 3.1.2 关键优化项

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 仿真房间分区效果差 | 分区算法主要适配传统扫地机地图 | 配合分区算法区分地图内部与外部逻辑，修改外围颜色为灰色(80,80,80) |
| 像素坐标到habitat坐标转换误差 | 生成障碍地图时会主动贴边（可视化优化） | 生成地图后主动裁切并上下翻转180度 |
| 走廊等狭长区域与房间相邻时无法区分 | 距离变换种子点在窄区域过少 | 新增 `_split_regions_by_narrow_connections` 后处理，提取core区域做竞争式生长分配 |
| 不同场景效果不稳定 | 参数和地图尺度强相关 | 引入MapConfig物理参数配置，所有阈值先用米/平方米定义，再按pixels_per_meter转换为像素 |

### 3.2 VLM房间类型推理

#### 3.2.1 位置判断方案

**方法一：四视角全图判断（当前机器人所在位置）**

输入四方向RGB图像，输出房间类型与置信度。测试结果表明对当前所在位置判断较为准确。

**方法二：四视角扇形区域判断**

输入四方向局部扇形区域RGB图像，输出每个方向的房间类型概率分布。对于部分场景（空旷区域、面向墙面）存在误判。

**最终策略**：两种方法结合使用，方法一保证当前位置准确性，方法二补充未到达区域的判断。

#### 3.2.2 Prompt设计

**位置判断Prompt核心要素**：

- 角色设定：机器人环境识别专家
- 输入说明：四方向RGB图像
- 判定维度：房间类型（客厅/卧室/厨房/卫生间/书房/阳台等）+ 房间概率（0-1）
- 输出格式：严格JSON格式

**扇形区域判断Prompt核心要素**：

- 角色设定：家庭服务机器人
- 输入说明：四方向扇形局部区域RGB图像
- 分析维度：家具、设施、空间布局、材质和环境
- 房间类型范围：living_room / kitchen / corridor / bedroom / bathroom / other
- 输出约束：每方向概率和为1，JSON格式

### 3.3 语义分区合并

基于VLM推理结果，对几何分区进行语义层面的优化：

```
输入：几何分区rooms + partition_map + semantic objects
    |
    v
_assign_objects_to_regions（将对象映射到对应region）
    |
    v
逐个region计算object_prior（根据对象类别统计房间先验分数）
    |
    v
是否需要分类器？ --> 是 --> 原始region分类（最大内切圆圆心 -> 采集4张图 -> classify_room）
    |（否）
    v
_fuse_room_predictions（融合object_prior与classifier）
    |
    v
_merge_adjacent_semantic_regions（合并相邻且同语义的原始region）
    |
    v
_refine_merged_regions（对merged mask再取最大内切圆，再次采集4张图进行分类）
    |
    v
回写merged语义到原始regions并更新area字段
    |
    v
生成可视化 + 保存结果（room_semantics JSON / PNG）
```

## 4. 深度学习分区方案

### 4.1 架构设计

采用两阶段U-Net架构：

| 阶段 | 模型 | 输入 | 输出 | 损失函数 |
|------|------|------|------|----------|
| 地图美化 | U-Net | 带障碍原始地图 | 清洁户型图（墙体/室外/室内） | CrossEntropy + Dice |
| 地图分区 | 多头U-Net | 美化后户型图 | boundary / seed / inside 三通道 | CrossEntropy + Dice |

### 4.2 数据定义

**地图美化标签**：

- 墙体 mask：`without_obstacles == 0`（黑色表示结构墙体）
- 室外 mask：`without_obstacles == 80`（未进入区域）
- 室内 mask：`without_obstacles == 255`（去掉家具杂点后的人可通行区域）
- 障碍物 mask：`(with_obstacles == 0) AND (without_obstacles != 0)`
- 恢复室内区域 mask：`(with_obstacles == 0) AND (without_obstacles == 255)`

**地图分区标签**：

- inside：室内可分区区域，由clean floorplan中值为255的像素生成
- boundary：不同房间之间的分界线，由room_ids图中相邻不同房间编号的交界位置计算
- seed：每个房间的中心区域，由room_ids对每个房间掩码进行形态学腐蚀后生成

### 4.3 性能指标

| 模块 | 单张推理时间 | CPU推理占用 |
|------|----------|----------|
| 地图美化 | 0.169s | 748.45 MB |
| 地图分区 | 2.408s | 776.05 MB |

### 4.4 优缺点分析

**优点**：

1. 基于U-Net结构实现，代码依赖简单，PyTorch环境即可运行
2. 输入为统一编码后的地图数据，不同地图来源间迁移较方便
3. 数据准备、训练、推理拆成独立脚本，目录结构清晰

**缺点**：

1. 依赖固定的像素语义约定，换地图编码规则需重写数据预处理
2. 分区阶段依赖手工构建的数据组织方式（room_ids / boundaries / seeds / inside_masks）
3. 后处理参数与地图尺度强相关，换尺寸时需重新调整
4. 新数据上表现不稳定时需继续补标，短期仍是"人工驱动型"迭代

### 4.5 总结

当前方法仍是传统几何方法的"变种"，虽在过程中不再依赖纯几何方法的计算，但输入部分仍是传统几何方法的地图结构。优点是只需前端地图形式确定，通过仿真环境即可初步构建数据集进行训练得到初代模型并进行后续迭代。

## 5. 去几何地图化分区方案（前瞻）

### 5.1 方案定位

与传统度量地图或三维重建方案不同，本方案关注高层语义结构，核心输出不是稠密点云或精确网格，而是相对稀疏的房间及物品语义的拓扑关系：

- 房间级语义节点
- 房间间连通关系
- 房间内显著物体语义
- 房间内物品位置

### 5.2 工作流程

```
机器人执行导航动作，系统持续获取RGB与位姿
    |
    v
满足采样条件时，在当前位置采集图像
    |
    v
将图像和约束式提示词发送至小型VLM
    |
    v
VLM返回房间类型、物品等结构化语义结果
    |
    v
基于"位姿邻近性 + 语义相似性"判断当前观测应归属哪个房间
    |
    v
匹配到已有房间？ --> 是 --> 更新已有房间节点，累计房间与物品语义统计
                   --> 否 --> 新建房间节点，记录当前房间与物品语义
    |
    v
当系统判断发生房间切换时，建立房间间连通边
    |
    v
将结果写入图结构、轨迹和观测日志，持续在线更新
```

### 5.3 数据输入输出

- **输入**：RGB数据 + 定位数据（绝对位置精度要求低，相对位置精度要求稍高）
- **输出**：语义拓扑图（JSON格式），包含房间节点、物品节点、邻接关系

### 5.4 优缺点

**优点**：

1. 不依赖完整SLAM管线，工程复杂度较低，适合原型验证和快速迭代
2. 在线增量式构建，天然适合在线导航、交互式探索和增量式场景理解任务
3. "房间+物品+连通关系"表达方式更贴近任务层需求，便于下游系统直接利用

**缺点**：

1. 房间类型识别、物品语义判断依赖模型输出质量，鲁棒性和场景覆盖能力直接影响拓扑图质量
2. 缺乏几何地图支撑，在用户表现体验上存在落差

## 6. 家庭语义记忆系统

### 6.1 系统定位

结合传统几何美化分区方案的前期使用验证效果、以及去几何地图分区方案的初步构思，设计一个分层式的家庭语义存储系统。

### 6.2 分层架构

| 层级 | 名称 | 定位 | 写入频率 | 生命周期 |
|------|------|------|----------|----------|
| L1 | 语义结构层 | 家庭空间的长期稳定骨架 | 低频 | 长期有效 |
| L2 | 动态事件层 | 高频观测与状态变化记录 | 高频 | 按时间窗口保留 |
| L3 | 记忆服务层 | L1+L2的翻译与按需组装 | 查询时生成 | 不持久化 |
| L4 | 用户展示层 | 面向用户的可理解视图 | 由L3驱动 | 实时生成 |

### 6.3 L1 - 语义结构层

**核心数据结构**：

**房间节点（Room Node）**：

```json
{
    "id": "room_001",
    "type": "living_room",
    "neighbors": ["room_002", "room_003"],
    "bbox": { "cx": 3.2, "cy": 6.1, "w": 5.0, "h": 4.0 },
    "last_confirmed": "2026-03-28T10:00:00Z"
}
```

**区域节点（Zone Node）**：

```json
{
    "id": "zone_sofa_area",
    "parent_room": "room_001",
    "bbox": { "cx": 2.1, "cy": 5.8, "w": 2.5, "h": 1.8 },
    "anchor_objects": ["obj_sofa_001", "obj_coffee_table_001"],
    "remembered_objects": [
        { "category": "remote_control", "evidence_count": 8 },
        { "category": "pillow", "evidence_count": 5 }
    ]
}
```

**锚点对象节点（Anchor Object Node）**：门、沙发、茶几、电视柜、冰箱、餐桌、床、柜子等大型不易移动家具。

```json
{
    "id": "obj_sofa_001",
    "category": "sofa",
    "canonical_room": "room_001",
    "canonical_zone": "zone_sofa_area",
    "bbox": { "cx": 2.1, "cy": 5.8, "w": 2.5, "h": 1.8 },
    "position_relative": "靠近西墙窗边"
}
```

**可移动对象记忆节点（Movable Object Memory Node）**：

```json
{
    "id": "obj_medicine_box",
    "category": "medicine_box",
    "habitual_locations": [
        { "room": "room_004", "evidence_count": 12 },
        { "room": "room_005", "evidence_count": 6 }
    ],
    "last_confirmed_location": {
        "room": "room_005",
        "source": "vlm"
    },
    "last_confirmed_at": "2026-03-28T20:15:00Z"
}
```

**设计要点**：

- L1只存节点与直接归属关系（对象->区域、区域->房间、房间->邻接房间）
- bbox一旦确认即冻结，区域bbox由多锚点观测推断稳定后确认，房间bbox由多次观测稳定后确认
- 节点不随单次观测失效，只有累计多次观测触发沉淀或结构变化才更新L1
- L1记录的是长期语义事实，不直接吞并原始观测噪声

### 6.4 L2 - 动态事件层

**定位**：记录高频、短生命周期的观测事件和状态变化，是"最近发生什么"的真实来源。

**写入来源**：VLM每次采样的原始输出；机器人运动轨迹；传感器触发事件。

**保留策略**：按时间窗口保留（如最近7天详细，更早的压缩摘要），不无限积累。

**核心事件类型**：

| 事件类型 | 说明 |
|----------|------|
| 观测事件（Observation Event） | 机器人在某区域的一次VLM采样结果，含可见物品全集、新增物品、消失物品 |
| 对象状态事件（Object State Event） | 记录某类可移动对象最近一次被观测到的房间 |
| 空间切换事件（Room Transition Event） | 记录机器人从一个房间进入另一个房间的时刻 |

**L2->L1沉淀规则**：

- **区域物品记忆沉淀**：某对象在某区域观测事件中多次出现（evidence_count超过阈值）-> 加入L1 Zone Node的remembered_objects
- **可移动对象位置更新**：某对象状态事件在某房间累计出现N次 -> 更新L1可移动对象habitual_locations分布
- **新房间发现**：多次未匹配已有房间的观测 -> 触发L1新节点创建
- **邻接关系强化**：空间切换事件多次重复 -> 强化L1 neighbors关系
- **遗忘衰减**：L1可移动对象长期无L2证据支撑 -> evidence_count自然下降

### 6.5 L3 - 记忆服务层

**定位**：不作为事实源，负责把L1+L2转化为不同消费者（任务执行、LLM、推理引擎）可用的形态。首先是"翻译层"，必要时允许存在可重建的派生索引。

**核心职责**：按需组装，而不是预先把所有视图都写死。

**任务场景**：

| 任务 | 输入 | 输出 | 推导来源 |
|------|------|------|----------|
| 找物（Object Search） | 目标对象类别 | 优先探索区域有序列表 + 各区域历史概率 | L1 habitual_locations + L1 Zone remembered_objects + L2最近对象状态事件 |
| 找人（Person Search） | 目标人物或全局搜索 | 按历史活动频率排序的房间列表 + 最近观测时间 | L2观测事件中visible_objects（含人物类别）按区域/时间聚合统计 |
| 巡逻异常检测（Patrol Anomaly Detection） | 当次巡逻各区域的观测事件 | 与基线偏差显著的区域 + 异常描述 | L2当次观测事件（new_objects / unseen_objects）vs L1 Zone remembered_objects基线 + not_expected_in语义规则 |

### 6.6 L4 - 用户展示层

**定位**：由L3驱动生成，面向用户可理解性设计，不直接暴露L1/L2的数据结构。

**核心原则**：用户要的是"看懂这个家"，不是"看懂系统内部建模"。

**三个展示视图**：

| 视图 | 驱动数据 | 展示内容 | 形式 |
|------|----------|----------|------|
| 房间总览视图（主视图） | L1房间节点 + 邻接关系 | 房间列表、连接关系、类型标签、区域节点 | 卡片列表 + 关系连线（语义关系图） |
| 语义布局草图（地图视图） | L1关系约束 | 房间相对位置、大小关系、连接点 | 生成式草图（只画相对关系，不画绝对坐标） |
| 房间详情视图（点击展开） | L1对象节点 + L2最近事件 | 常见家具、高频物品、状态变化、置信度、用户纠错入口 | 详情卡片 |

## 7. 技术路线对比与选型建议

| 维度 | 传统几何分区+VLM语义 | 深度学习分区 | 去几何地图化分区 |
|------|------------------|----------|--------------|
| 地图依赖 | 强依赖障碍地图 | 强依赖障碍地图 | 仅需RGB+粗定位 |
| 分区精度 | 中等，参数敏感 | 中高，依赖数据量 | 无几何边界，语义级别 |
| 语义能力 | VLM推理（准确率高） | 无（仅分区） | VLM推理（准确率高） |
| 端侧部署 | 分区算法轻量，VLM需推理资源 | U-Net轻量（CPU可跑） | VLM需推理资源 |
| 场景适配 | 需按场景调参 | 需按场景补数据 | 场景泛化性好 |
| 工程复杂度 | 中等 | 中等（需标注数据） | 较低 |
| 成熟度 | 已验证可行 | 初步验证 | 概念阶段 |

**当前推荐路线**：短期以"深度学习分区 + VLM语义标注"为主线推进，同步探索去几何地图化分区方案作为中长期技术储备。

## 8. 里程碑规划

### 第一阶段：几何分区算法稳定化

- 完成CW分区算法在habitatsim自采数据集上的参数优化与效果验收
- 解决走廊/狭长区域分区问题（narrow connection splitting）
- 输出：稳定的几何分区模块，适配多种户型

### 第二阶段：深度学习分区模型迭代

- 扩充地图美化与分区的训练数据集（含国内真实场景数据）
- 优化后处理流程的鲁棒性
- 输出：可部署的两阶段U-Net分区模型

### 第三阶段：VLM语义标注与合并优化

- 使用habitatsim+自采数据进行房间类型判断测试与prompt优化
- 优化分区合并及房间语义提取效果
- 输出：完整的几何分区+语义标注pipeline

### 第四阶段：语义记忆系统搭建

- 实现L1/L2核心数据结构与沉淀规则
- 实现L3记忆服务层的找物/找人/异常检测核心接口
- 与VLN推理流程集成

### 第五阶段：去几何地图化方案验证

- 在线语义拓扑建图原型验证
- 与几何分区方案的效果对比
- 输出：去几何化方案可行性报告

## 9. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| VLM推理延迟过高 | 影响实时性 | 优先使用8B/30B-A3B等小模型；批量推理优化 |
| 国内场景与仿真差异大 | 分区/语义效果降级 | 采集国内真实场景数据补充训练集 |
| 深度学习模型泛化性不足 | 换场景需重新标注 | 持续补标+数据增强策略 |
| 地图编码格式变更 | 数据预处理链路失效 | 抽象地图编码接口，降低耦合 |
| 语义记忆噪声积累 | L1数据质量下降 | L2->L1沉淀设置证据阈值，遗忘衰减机制 |

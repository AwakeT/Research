# 守护程序架构与设计方案

> **来源文档**：方案设计文档/守护程序架构图.pdf + 方案设计文档/守护程序设计方案.pdf
> **评审维度**：系统和方案设计
> **更新时间**：2026-04

## 一、设计背景与目标

Kinbot机器人系统由多个ROS2节点组成（导航、决策、雷达、SLAM、探索、地图美化/分区等），各节点崩溃、假死或闪退会导致系统不可用。需要设计一个守护程序，在后台长期运行、无需用户干预即可持续保障系统健康运行。

**守护程序的核心使命**：让系统或服务始终健康地活着。

## 二、整体架构

守护程序包含三大功能模块：

```
守护程序
├── 节点监控（每2s遍历一次）
├── log删除（每24h/每7天扫描一次）
└── 闲时重启（定时任务）
```

## 三、节点监控模块

### 3.1 监控节点列表设计

采用`std::vector<ProcessInfo>`维护监控节点列表，每个节点包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 节点名称 | 进程标识 | nav_user_node |
| 运行路径 | 节点在系统中的路径 | /app/nav_app/install/nav_user/lib/nav_user |
| 启动脚本路径 | 启动脚本所在目录 | /app/nav_app/install/nav_user/share/nav_user/auto_sh |
| 脚本名称 | 启动命令 | start_nav_user |
| 需要参数 | 脚本启动是否需要参数补充 | false |

**当前监控节点**：
1. nav_user_node — 导航用户节点
2. decision_node — 决策节点
3. ldlidar — 雷达节点
4. cartographer_node — SLAM建图节点
5. manager_node — Nav2节点管理器
6. exploration_node — RRT探索节点
7. cw_map_foxy_node — 地图美化节点
8. cw_map_part_node — 地图分区节点

### 3.2 监控流程

```
大导航节点是否启动？
├── 否 → 启动未正常开启节点
└── 是 → tc.nav_ready = 1
         ↓
     是否存在节点异常崩溃？（每2s遍历一次）
     ├── 是 → 启动节点
     └── 否 → 继续监控
```

### 3.3 可扩展性设计

列表结构对类似的C++应用场景具有良好的可扩展性，可按需进行节点替换或新增，无需修改监控逻辑本身。

## 四、日志清理模块

### 4.1 清理策略

针对log文件设计了三层递进清理规则：

```
log文件最终修改时间 > 30天？
├── 是 → 删除不合规文件
└── 否 → log文件数量 > 50个？
         ├── 是 → 删除不合规文件（时间排序）
         └── 否 → log文件夹大小 > 50MB？
                  ├── 是 → 删除不合规文件（时间排序）
                  └── 否 → 跳过
```

**触发条件**：
- userdata占用 > 80% → 即时触发删除相关路径下的log文件
- 否则 → 每24小时检查一次

**扫描周期**：每七天执行一次全量扫描

### 4.2 清理规则细节

1. 按照文件修改时间进行检查：把当前历史最悠远的、单个log文件中性能最大的、最终路径中文件数量最多的，先进行计算排列
2. 为每个log文件夹中部分空间预留为log存储，一次性清除log文件后，后续留作foxy文件来源

## 五、闲时重启模块

### 5.1 重启条件

```
当前节点运行时间 > 7天？
├── 是 → 判断闲时条件：
│        a. 有效时间段之内
│        b. 在充电桩上
│        c. nav_app系统占用率 < xx%
│        ↓
│    最终处于闲状态？
│    ├── 是 → 重启部分功能节点
│    └── 否 → 挂起
└── 否 → 继续监控
```

### 5.2 设计要点

- 检查节点连续运行时间是否超过7天作为重启触发条件
- 闲时判断需满足三个条件同时成立：时间段、充电状态、系统负载
- 仅重启部分功能节点，而非整机重启

## 六、守护程序的完整能力矩阵

| 能力类别 | 具体功能 |
|---------|---------|
| 生命周期与可用性管理 | 随系统启动/关闭自启自停；看门狗探测崩溃/假死/闪退并自动重启；单实例控制防重复启动；平滑重启/热升级 |
| 系统资源与性能管理 | CPU/内存/磁盘/网络实时采样与上限保护；日志滚动/缓存清理/过期文件删除；动态调整线程/进程池 |
| 配置、策略与状态同步 | 启动时读取配置，运行时支持动态热加载；通过D-Bus/消息总线/共享内存下发策略；故障转移 |
| 监控、日志与告警 | stdout/stderr重定向到集中日志；分级输出（DEBUG/INFO/WARN/ERROR）；阈值告警 |
| 安全与访问控制 | 最小权限用户运行；Gatekeeper/代理模式鉴权限流；审计与追踪 |

## 七、设计挑战与创新点

1. **基于列表的可扩展节点监控**：ProcessInfo结构化设计，新增/替换监控节点仅需修改列表，无需改动监控逻辑
2. **三层递进日志清理策略**：按时间→数量→体积三维度逐层清理，平衡存储空间与日志保留需求
3. **多条件闲时重启**：融合运行时长、充电状态、系统负载三重条件判断，避免在工作状态下执行重启

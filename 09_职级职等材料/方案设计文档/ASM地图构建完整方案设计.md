# ASM地图构建完整方案设计

> Annotated Semantic Map (ASM) —— 标注语义地图构建方案
>
> 版本：v1.0 | 日期：2026-04-22

---

## 一、方案概述

### 1.1 背景与目标

ASM（Annotated Semantic Map，标注语义地图）是面向室内视觉语言导航（VLN）任务的核心地图表示。其核心目标是将机器人在室内环境中感知到的**RGB图像**、**深度信息**和**位姿数据**融合为一张包含障碍物信息和语义标注的鸟瞰地图，为下游VLN推理模块提供结构化的环境理解。

ASM地图构建系统基于MapNav论文框架演进而来，主要改进包括：
- 将语义分割方案从Mask2Former替换为**Grounding-DINO + MobileSAM**组合，支持开放词汇检测
- 引入**JSON/DB双存储**机制，分离实时查询与历史溯源
- 设计**动态地图更新策略**，支持场景变化后的地图适配
- 适配**JQR设备**，支持真机部署

### 1.2 系统输入输出

| 维度 | 内容 |
|------|------|
| **输入** | RGB图像、深度图像（RGBD相机）、机器人位姿（SLAM/轮式里程计） |
| **输出** | ASM地图 = 障碍物栅格地图（鸟瞰图） + 语义标注信息（物品名称、位置、置信度） + Agent轨迹 |

## 二、系统总体架构

ASM地图构建系统采用**四层架构**设计：

```
┌─────────────────────────────────────────────────────────────────┐
│                         输入层                                   │
│         深度信息        机器人位姿        RGB图像                  │
└────────┬──────────────────┬──────────────────┬──────────────────┘
         │                  │                  │
         │                  │                  ▼
         │                  │         ┌─────────────────┐
         │                  │         │   目标检测器      │
         │                  │         │ (Grounding-DINO) │
         │                  │         └────────┬────────┘
         │                  │                  ▼
         │                  │         ┌─────────────────┐
         │                  │         │   图像分割        │  语义提取层
         │                  │         │  (MobileSAM)     │
         │                  │         └────────┬────────┘
         │                  │                  │
         ▼                  ▼                  ▼
┌─────────────┐  ┌──────────────┐  ┌──────────────────┐
│ 障碍地图生成  │  │ Agent轨迹生成 │  │  目标物坐标计算    │  鸟瞰投影层
│(ObstacleMap)│  │              │  │ (2D→3D→世界坐标)  │
└──────┬──────┘  └──────┬───────┘  └────────┬─────────┘
       │                │                   │
       ▼                ▼                   ▼
┌─────────────────────────────────────────────────────────────────┐
│                       模态融合层                                  │
│                       ASM地图                                    │
│          障碍栅格 + 语义标注 + Agent轨迹 → 最终输出               │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 架构演进（JQR适配）

修改前架构中，障碍地图由深度信息直接生成；修改后架构中：
- **障碍地图**改为使用导航模块提供的现有地图，降低自建障碍地图的累积误差
- **语义信息存储**新增DB存储通道，支持存在状态跟踪、历史记录保留、`last_show_time`字段
- **信息结构**扩展为：序列ID、物品名称、时间戳、存在状态、2D像素坐标、世界坐标

## 三、RGB数据处理流程（语义提取）

### 3.1 对象检测阶段

采用 **Grounding-DINO** 作为开放词汇目标检测器：

```
RGB图像
  │
  ▼
Grounding-DINO（目标检测）
  │── 检测框（bounding box）
  │── 语义标签（物体类别）
  │── 置信度分数
  ▼
MobileSAM（实例分割）
  │── 对象掩码（mask）
  ▼
检测结果输出
```

**核心函数**: `detect_and_update_objects()`

1. RGB图像输入至Grounding-DINO，获取检测框和语义标签
2. 对每个检测到的对象，通过MobileSAM生成实例分割掩码
3. 输出包含类别、置信度、边界框、掩码的完整检测结果

### 3.2 检测类别配置

当前识别类别经过优化精简（去除相似类避免注意力分散）：

```
chair, table, bed, sofa, plant, door, window, tv, monitor, desk, cabinet
```

> **优化说明**: Grounding-DINO使用注意力机制关联文本提示和图像特征，过多相似的文本标签（如plant与potted plant、sofa与couch）会分散模型注意力。精简后的类别列表使模型能更专注地学习每个类别的独特特征，检测效果显著提升。

### 3.3 语义地图更新

**核心函数**: `ValueMap.add_object()`

```
对象信息（像素坐标） + 相机内参 + 位姿（外参）
  │
  ▼
将2D掩码投影到3D空间（反投影）
  │
  ▼
转换到地图坐标系（世界坐标 → 栅格坐标）
  │
  ▼
更新语义地图的对应区域
  │
  ▼
记录对象位置和属性
```

### 3.4 检测结果持久化

| 存储方式 | 内容 | 用途 |
|---------|------|------|
| **JSON文件** | 序列ID、物品名称、时间戳、2D像素坐标、世界坐标 | 实时查询（覆盖式更新） |
| **DB存储** | 序列ID、物品名称、时间戳、存在状态、2D像素坐标、世界坐标、last_show_time | 历史溯源（保留式更新） |
| **标注图像** | 带检测框的RGB图像 | 调试与可视化 |

## 四、深度数据处理流程（障碍地图）

### 4.1 深度数据预处理

**核心函数**: `update_map()`

```
原始深度图像
  │
  ▼
深度值裁剪（过滤超出有效范围的值）
  │
  ▼
深度数据验证（有效像素比例检查）
  │
  ▼
深度归一化（除以最大深度值）
  │
  ▼
转换为ObstacleMap输入
```

### 4.2 几何信息提取

- **有效深度检测**: 过滤掉无效或超出范围的深度值
- **相机参数应用**: 利用内参矩阵进行2D→3D反投影
- **位姿变换**: 利用外参矩阵将相机坐标系转换到世界坐标系

### 4.3 障碍物地图生成

**核心函数**: `ObstacleMap.update_map()`

```
深度图像 + 相机位姿
  │
  ▼
深度值转换为3D点云（利用相机内参）
  │
  ▼
应用相机外参变换（相机坐标 → 世界坐标）
  │
  ▼
投影到鸟瞰图坐标系（3D → 2D俯视）
  │
  ▼
更新障碍物占用栅格
  │
  ▼
更新可通行区域信息
```

**障碍地图数据流**:

```
深度信息 ──┐
           ├── 对齐时间戳 ──→ 障碍地图更新(10Hz) ←── 相机参数（内参+外参）
定位信息 ──┘
```

**语义信息数据流**:

```
RGB信息 ──→ 目标检测器 ──┐
深度信息 ──┐              │
障碍地图 ──┤              ▼
           ├── 3D坐标→2D像素坐标 ←── 相机参数（内参+外参）
           │         │
           │         ▼
           │    语义信息(2Hz)
```

### 4.4 轨迹记录

记录Agent在扫描建图过程中的移动路径，用于：
- ASM地图上的历史轨迹可视化
- 评估扫描覆盖率
- 辅助路径规划

## 五、实时更新闭环（软件时序）

### 5.1 ASM一次实时更新闭环

系统涉及的核心组件：**ASMSystem**、**ZEDDataSubscriber**、**PoseBuffer**、**ObstacleMap**、**ValueMap**、**RealObjectDetector**

#### 阶段一：传感器数据到达（回调）

```
ZEDDataSubscriber:
  depth_callback()        // 含深度有效性检查
    → 取 latest_pose
    → interpolate_T_g_b(depth_ts)  // 位姿插值
    → 生成 depth-pose pair
    → new_pair_ready = True

  on_new_depth_pose_pair()
    → new_pair_pending = True
```

#### 阶段二：主循环处理

```
ASMSystem.run_mapping_system() / spin_once():
  if new_pair_pending == True:
    → update_map_realtime()
    → get_latest_depth_pose_pair()  // {depth, rgb, position, yaw, ...}
    → 移动/旋转阈值检查
    → create_camera_to_world...()
    → update_map(depth, T_cam_world)  // 更新占据栅格+轨迹

    [每5步做一次目标检测]  // opt
      → detect(rgb)
      → detections
      → [每个检测框] add_object(...)

    → _publish_occupancy_grid()
    → new_pair_pending = False
```

#### 阶段三：定期落盘与监控

```
if step % 10 == 0:
  → cv2.imwrite(map.png)
  → save_performance_stats()
```

## 六、语义信息更新策略

### 6.1 首次观测流程

```
获取RGB+Depth（基于最大有效距离）
  │
  ▼
目标检测、分割、点云提取
  │
  ▼
获取物品坐标信息
  │
  ▼
是否为新目标物（类别）？ ──是──→ 更新物品信息、时间戳至json（覆盖式）
                                  更新物品信息、时间戳至db（保留式）
```

### 6.2 持续维护流程

```
物品是否被检测（基于最佳检测距离3m）？
  │
  ├─ 是 ──→ 评分++
  │
  └─ 否 ──→ 物品评分--
              │
              ▼
          物品评分<XX？ ──是──→ 删除语义信息
              │
              └─ 否
              │
              ▼
          位置变动>50cm？ ──是──→ 更新物品信息、时间戳至json（覆盖式）
              │                    更新物品信息、时间戳至db（保留式）
              └─ 否 ──→ 返回循环
```

**核心机制说明**:
- **评分机制**: 每次检测到物品则评分递增，未检测到则递减，评分低于阈值则判定物品已消失并删除
- **位置变动检测**: 物品世界坐标变动超过50cm时触发位置更新
- **双写策略**: JSON覆盖式更新保证实时性，DB保留式更新保证历史可追溯

## 七、动态地图更新方案设计

### 7.1 现状分析

当前ASM地图在更新过程中**仅可在首次构建时**进行障碍地图与语义地图的更新（追加式，不会抹去历史信息），未考虑后续场景发生较大变化后原先ASM地图无法适配后续任务的情况。

### 7.2 候选方案对比

#### 方案A：基于Agent半径的地图更新

**原理**: 以传感器最大可视范围与Agent物理半径作为空间分层依据。

| 层级 | 范围 | 策略 |
|------|------|------|
| 主图（长期记忆） | 3m圈 | 追加式更新，不删除历史，保证全局一致性 |
| 子图（短时记忆） | 1m圈 | 每帧先清空再重画，反映当前瞬间局部环境 |

**同步流程**: 匹配 → 融合 → 写回（将子图可信变化同步到主图）

**存在问题**:
- 需要维护两张地图 + 一次双向同步，工程实现复杂度翻倍
- 匹配-融合阶段若位姿漂移，会把错误写回主图，形成障碍地图错位及语义信息错误

#### 方案B：基于场景变化的地图更新

**原理**: 将初始可见范围的障碍+语义存为"参考帧"，按区域划分，后续帧与参考帧做变化评分。

**流程**:
1. 首次建图后生成"参考帧快照"，按区域划分存储
2. 每来一帧，在同一范围内做"变化评分"
3. 分数超过阈值 → 整个范围标记为"待更新"，用新观测替换旧内容
4. 分数未超阈值 → 跳过，保持主图不动

**存在问题**:
- 对评价函数的设计要求较高
- 可能出现语义无变化但障碍地图变化大的情况（如家具移动但类别不变）

#### 方案C：基于任务预测的地图更新（推荐）

**原理**: 结合VLN推理的目标点，沿最短路径选择性更新。

**流程**:
1. VLN每次推理完成后提供一个目标点
2. 计算Agent到目标点的最短路径
3. 结合机器人正方向视场（锥形）绘制更新范围
4. 仅在该范围内进行障碍地图和语义信息更新

**优势**: 按需更新，计算量可控，与任务紧密耦合

### 7.3 方案选型建议

| 维度 | 方案A（Agent半径） | 方案B（场景变化） | 方案C（任务预测） |
|------|:---:|:---:|:---:|
| 工程复杂度 | 高 | 中 | 低 |
| 实时性 | 高 | 中 | 中 |
| 鲁棒性 | 低（位姿漂移敏感） | 中（评价函数依赖） | 高 |
| 计算开销 | 高（双图维护） | 中 | 低（按需更新） |
| 与VLN耦合 | 无 | 无 | 强 |

**推荐方案C**，辅以方案B的评分机制作为语义信息维护策略。

## 八、语义识别与分割方案选型

### 8.1 当前方案：Grounding-DINO + MobileSAM

| 环节 | 模型 | 说明 |
|------|------|------|
| 目标检测 | Grounding-DINO | 开放词汇检测，支持自然语言文本提示，显存≥8GB |
| 实例分割 | MobileSAM | 轻量化分割，模型体积10MB，推理速度约10帧/s，显存100~300MB |

### 8.2 方案调研对比

| 工作 | 所用模型 | 物品识别方法 |
|------|---------|------------|
| VLFM | YOLOv7 + Grounding-DINO + Mobile-SAM | COCO数据集物品用YOLOv7，其余用Grounding-DINO补充识别 |
| unigoal | Grounded-SAM | 识别图像中的所有对象，然后提示VLM |
| MapNav | Mask2Former | 分割物体，对关键区域添加显式文本标签 |
| InstructNav | GLEE | 对RGB图像进行语义分割（开源） |

### 8.3 模型详细对比

| 模型 | 优点 | 缺点 | 显存需求 |
|------|------|------|---------|
| **YOLOv7** | 算力要求低、模型小、识别速度快；COCO数据集精度高；可自行增加识别目标 | 新增目标需大量标注数据；小目标检测效果较差 | 1~2GB |
| **Grounding-DINO** | 语言驱动动态识别，适合VLN场景；支持图像和文本同时推理；无需额外训练 | 模型大、推理速度慢；微调难度大 | ≥8GB |
| **Mobile-SAM** | 体积小(10MB)、推理速度快(~10帧/s)；保留SAM核心分割能力 | 细小物品分割能力有限；低质量图像分割能力受限 | 100~300MB |
| **Grounded-SAM** | 支持开放词汇检测与分割；分割精度高；可自动标注 | 计算成本高、推理速度慢 | ≥8GB |
| **Mask2Former** | 分割精度高；同时处理语义/实例/全景分割 | 仅支持COCO(80类)+ADE20K(150类)；不支持开放词汇 | ≥16GB |
| **GLEE** | 结合CLIP视觉-语言模型；支持开放词汇 | 显存需求极高 | ≥24GB |

### 8.4 当前方案已知问题与优化方向

**检测问题**:
- Grounding-DINO在实机环境中对小物品识别不足（如花瓶、水杯）
- 部分场景存在误检（窗帘识别为门、投影识别为窗户等）
- 不同相机高度（1.1m vs 1.5m）识别效率不一致

**分割问题**:
- MobileSAM对小物品、交杂在一起的物品分割效果较差
- 大物品（沙发、床）分割效果尚可

**优化方向**:
1. 精简类别过滤器，去除相似类别（已验证有效）
2. 考虑更换MobileSAM为完整SAM或其他分割方案以提升分割质量
3. 实机场景可设置多高度RGB相机并融合识别结果

## 十、硬件需求与相机参数

### 10.1 相机参数

| 参数 | MapNav (D435i) | Navid/Uni-Navid (D455) |
|------|----------------|------------------------|
| **RGB分辨率** | 640×480, FOV 90° | 640×480, FOV 90° |
| **深度分辨率** | 1280×720 @90fps | 1280×720 @90fps |
| **RGB分辨率（原生）** | 1920×1080 @30fps | 1280×800 @30fps |
| **深度范围** | 0.2~10m（推荐0.3~3m） | 0.4~10m（推荐0.4~6m） |
| **视场角(FOV)** | 87°×58°（深度） | 86°×57° |
| **IMU** | BMI055（6轴） | BMI055（6轴） |
| **模型输入分辨率** | 384×384 | 640×480 |

### 10.2 相机选型说明

- **RGB输入**: 640×480分辨率在仿真场景中已可完成基础识别任务
- **深度信息**: 仿真中给出的深度距离与实际D435i可信距离差距较大；RGB与深度信息输入源不强行要求同一设备，可设置多个深度信息输入源后期拼接
- **多高度方案**: 不同高度的相机识别效果不一致，实机中可设置多高度RGB相机并融合结果

### 10.3 算力需求

| 组件 | 需求 |
|------|------|
| **VLN推理模型** | 基座模型 Qwen2.5vl-7b，加载及推理显存最大17GB |
| **推理延迟** | 单图200-300ms，4张图片（1ASM+1当前帧+2历史帧）延时900-1000ms |
| **额外图片代价** | 每增加一张图片推理时长增多200-250ms |
| **Grounding-DINO** | 显存 ≥8GB |
| **MobileSAM** | 显存 100~300MB |

## 十三、总结

ASM地图构建系统通过 **Grounding-DINO + MobileSAM** 的开放词汇检测-分割组合，结合深度图像的障碍物栅格构建，实现了支持VLN任务的标注语义地图生成。当前系统已完成基本建图功能验证，下一步重点工作包括：

1. **动态地图更新**: 采用基于任务预测的更新方案（方案C），辅以评分机制维护语义信息
2. **检测分割优化**: 精简类别过滤器、评估更优分割模型、适配多场景
3. **工程优化**: JSON→msgpack加速、障碍地图源切换、坐标系统一
4. **真机部署**: JQR设备适配、多高度相机融合、SLAM位姿集成
